# Task detail moves from a right-anchored drawer to a blocking centered modal with an unsaved-edits close-guard

The Task detail editor is now a **centered modal `Dialog`** (MUI `Dialog`) instead of the right-anchored `Drawer` it used to be. On desktop it opens centered at ~920px wide, capped at ~85vh, with two internally-scrolling columns: task fields plus the label picker on the left over a pinned footer (Save / Cancel and a visually separated, error-styled Delete Task), and the task notes as an independently scrollable **chat** on the right — oldest-first, newest at the bottom, auto-scrolling to the newest note on open and after each add, with an icon send button where Enter inserts a newline and never sends. Below the `sm` breakpoint the dialog goes fullscreen and the two columns stack (fields first, notes below).

Dismissal is **guarded**: a backdrop click or Esc is ignored while there are unsaved field edits, and closes the dialog when there are none. The ✕ icon and the Cancel button **always** close, deliberately, discarding unsaved field edits. Saving persists the task and its label set together, **keeps the dialog open** with a brief "Saved" confirmation, and re-baselines the edit snapshot so the guard clears. Notes are persisted the moment they are sent, so they are never at risk from any close path.

Three pieces are built as reusable modules because the sibling Project and Customer detail tickets will reuse them: a generic centered-dialog shell (`components/DetailDialog.tsx`) with an optional right (notes) column and the close-guard wiring; a presentational chat-style notes panel (`components/NotesChatPanel.tsx`) that both Task (`Csa_notesService`) and Project (`Csa_projectnotesService`) containers can back with their own data seam; and a pure `isFormDirty(current, original)` predicate (`shared/formDirty.ts`) plus a pure `selectNotesOldestFirst` ordering (`data/notes.ts`), each unit-tested at the seam.

**Why:** the Task detail carries more than a drawer comfortably shows — all task fields, a label picker, and a running notes conversation. A narrow right rail forces the notes to sit *below* the fields, so reading history while editing means scrolling the whole rail; the pinned Save/Cancel/Delete actions also scroll out of reach. A centered two-column modal gives the fields and the notes their own scroll regions side by side and keeps the footer actions always visible, which matches the Planner-style "open a card" mental model the epic (#41) is after. Making the modal **blocking** — backdrop/Esc ignored while dirty — prevents the classic drawer foot-gun where a stray click on the page silently throws away unsaved edits; the ✕ and Cancel remain unguarded because those are *deliberate* discards. Save keeps the dialog open because editing a card is an iterative activity (tweak a field, add a note, tweak again); bouncing the user out on every Save would fight that loop. Ordering notes **oldest-first with the newest at the bottom** and auto-scrolling there makes the panel read like a chat, where the latest entry is where attention lands and where the composer sits — the opposite of the old newest-first timeline, which is the right convention for a bottom-anchored composer.

**Consequence:** there are now two structurally different detail surfaces in the app — the older right-anchored `Drawer` (still used by the Customer and Project panes until #43/#44 migrate them) and the new centered `DetailDialog`. That divergence is temporary and intentional: `DetailDialog`, `NotesChatPanel`, `isFormDirty`, and `selectNotesOldestFirst` are deliberately generic so the Project and Customer tickets adopt the same shell rather than growing a third pattern. `DetailDialog` treats its notes column as optional, so Customer (which has no notes) renders a single column through the same shell. The close-guard depends on an accurate dirty snapshot: the Task dialog captures the pristine field values *and* selected label ids on open and re-captures them after each successful Save, and `isFormDirty` compares label-id arrays order-independently so toggling a label off then on does not read as a pending edit. Because notes persist on send (not on the form's Save), they survive Cancel/✕/Esc regardless of the guard — the guard only ever protects the left-column *field* edits. `TASK_DRAWER_WIDTH` is retained in `shared/layout.ts` for the remaining drawers; the dialog's own sizing lives beside it as `DETAIL_DIALOG_WIDTH` / `DETAIL_DIALOG_MAX_HEIGHT`.
