# Project notes live in a dedicated ProjectNote table, separate from task Notes

Informational context recorded against a Project — an FYI, a recorded decision, a customer update swept from mail or Teams — is stored as a **Project Note** in its own `csa_projectnote` table (Text + a required Project lookup). This is distinct from a **Note**, which remains strictly task-scoped. Actionable items are still Tasks; only non-actionable context becomes a Project Note.

An earlier revision of this decision reparented the single Note record to attach to either a Task or a Project. That was reverted in favour of two separate tables.

**Why:** the app's data is the single source of truth for an external status workflow that sweeps email/Teams and creates Tasks for actionable work. Much of what it ingests is *not* actionable (FYIs, decisions, "the customer pushed the date") and needs a per-project activity ledger to summarize. Two options give the project a ledger: (a) dual-parent the one Note table, or (b) a dedicated ProjectNote table. We chose (b) — separate tables — because it keeps each record type single-purpose and single-parent (a Note always has a required Task; a Project Note always has a required Project), avoids the "exactly one of two optional lookups" invariant the app and every skill would otherwise enforce by hand, and lets the two ledgers evolve independently. The original glossary rule that Notes are task-scoped is preserved.

**Consequence:** there are now two note tables and two data seams (`notes.ts` for task Notes, `projectNotes.ts` for Project Notes). The project delete cascade removes the project's Project Notes (via `deleteProjectNotes` against `csa_projectnote`); task Notes still cascade with their Task. Reads and writes for the two never mix. The cost is a second table and seam to maintain; the benefit is a clean, hard-to-misuse parenting model. `CONTEXT.md` keeps **Note** task-scoped and adds **Project Note** as its own term.
