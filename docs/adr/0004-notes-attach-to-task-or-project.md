# Notes attach to a Task or a Project, not to Tasks only

A Note can now be parented by either a Task **or** a Project (exactly one of the two). Previously Notes were strictly task-scoped and "never reparented." Project Notes hold informational context — an FYI, a recorded decision, a customer update swept from mail or Teams — that is not itself a unit of work. Actionable items remain Tasks; only non-actionable context becomes a Project Note.

**Why:** the app's data is becoming the single source of truth for an external status workflow that sweeps email/Teams and creates Tasks for actionable work. Much of what it ingests is *not* actionable (FYIs, decisions, "the customer pushed the date"). With notes task-only, that context had nowhere to live without inventing fake Tasks and polluting the board. Letting a Note hang off a Project gives the status workflow a real per-project activity ledger to summarize, while keeping the board reserved for genuine work. This mirrors the prior markdown tracker, where sweeps wrote project-level notes and notes were the "how it got here" audit trail.

**Consequence:** this reverses the earlier glossary rule that Notes are task-scoped (`CONTEXT.md` updated). The Note record carries an optional Project lookup alongside its Task lookup, and code/queries that assumed a Note always has a Task parent must handle the Project-parented case. It is hard to reverse once the external skills begin writing Project Notes at volume, which is why it is recorded here. Cascade delete extends accordingly: deleting a Project takes its Project Notes with it (Task Notes still cascade via their Task).
