# Project contacts are surfaced as a full linked-records table with immediate writes, not a chip picker

On the Project detail pane, the Contacts linked to a Project (the project side of the Contact↔Project many-to-many) are shown as a **full table** — one row per linked Contact with all fields visible (name, role, email, phone) — and every mutation persists **immediately** (optimistic, revert on error). The row's destructive action is **Unlink** (detach the M:N association; the Contact keeps living under its Customer), never Delete. The pane offers two add paths above the table: link an existing Contact of the Project's Customer, and create a brand-new Contact that is parented to the Project's Customer and auto-linked in one action. Inline editing of a Contact's fields is allowed here and writes straight through to the shared Contact record.

This deliberately diverges from the Task pane's **Label** picker, which renders links as compact toggle chips and batches attach/detach with the pane's Save. The two many-to-many editors therefore look and behave differently on purpose.

**Why:** a Label is a bare name — a chip conveys everything there is to know, and batching on Save fits a lightweight tagging gesture. A Contact is a rich record (four fields) that the user needs to *read* while on the Project, so a table that shows every field is the honest surface. Once the surface is a table with per-row actions and inline create, direct/immediate writes read as ordinary table CRUD; deferring them to a form Save button would be surprising. Unlink-not-delete keeps the single-owner rule intact (a Contact belongs to exactly one Customer per `CONTEXT.md`) — removing a link from a Project must not destroy a Contact that other Projects or the Customer still use. Creating a Contact "from the Project" is a convenience only: the record is still owned by the Customer; the Project side just supplies the Customer id and auto-links.

**Consequence:** there are now two structurally different M:N editors in the app — chip-batched (labels) and table-immediate (project contacts) — and future readers must not assume one is the template for the other. A shared `ContactsTable` presentational component backs both the Customer contacts table (create/edit/delete) and the Project contacts table (link/create/edit/unlink); the two containers differ only in which row actions they enable and whether the destructive verb is Delete or Unlink. Immediate writes mean the Project pane mutates live data before its Save, so link/unlink/create take effect even if the user later cancels the form — accepted deliberately, matching the table CRUD mental model. The project↔contact association mechanics (`@odata.bind` PATCH) are exercised for real for the first time by this surface and must be verified against the live org.
